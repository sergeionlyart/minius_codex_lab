# Аудиторское заключение по `minius_codex_lab`

## Итоговый вердикт

**Статус на 11 июля 2026 года: CONDITIONAL NO-GO.**

Публичный репозиторий **архитектурно соответствует первоначальному замыслу и является сильной основой проекта**, однако выпуск `v1.0.0-beta.1` пока **нельзя рекомендовать широкому кругу внешних пользователей как готовую юридическую рабочую среду**.

Допустимый режим использования сейчас:

- разработка и code review;
- тестирование на синтетических данных;
- ограниченный технический пилот опытными пользователями;
- развитие скиллов, ролей и инструментов.

Пока не рекомендовано:

- начинать реальные юридические дела;
- загружать рабочие материалы органов государственной власти;
- использовать релиз для подготовки юридически значимых документов;
- рекламировать `v1.0.0-beta.1` как готовый пользовательский продукт.

Главная причина — **в опубликованном релизе восемь из двенадцати скиллов имеют невалидный YAML-frontmatter**, включая два ключевых скилла: `legal-monitoring` и `verifiable-legal-document`. При этом действующий валидатор не обнаруживает такую ошибку, поэтому зеленый CI создает ложное впечатление полной готовности.

Проверены:

- ветка `main`, соответствующая на момент проверки коммиту `25270c7`;

GitHub показывает успешные CI и release workflow. Однако сам репозиторий прямо предупреждает, что Codex CLI, hooks и rules не были фактически smoke-tested при первоначальной миграции.

Независимый локальный checkout, распаковку опубликованного ZIP и запуск Codex CLI в моей среде выполнить не удалось, поэтому это **глубокий статический и конфигурационный аудит**, но не эксплуатационная сертификация собранного архива. Настройки GitHub, которые не видны из публичного содержимого — например обязательность CI для `main`, branch protection и запрет force-push, — также считаются непроверенными.

## 1.2. Два разных `AGENTS.md` — верное решение

- запрет публикации реальных юридических данных;

# 2. Критические блокеры

## P0-1. Восемь из двенадцати скиллов имеют невалидный YAML-frontmatter

Ошибки присутствуют непосредственно в опубликованном теге `v1.0.0-beta.1`, а не только в текущей ветке.

Повреждены:

```text
matter-intake
ukrainian-legal-research
quantitative-impact-analysis
legal-monitoring
legal-qa-audit
redaction-and-disclosure
session-memory-handoff
verifiable-legal-document
```

GitHub для каждого файла показывает:

```text
Error in user YAML:
mapping values are not allowed in this context
```

Причина — двоеточие с последующим пробелом внутри некавыченного значения `description:`:

```yaml
description: Проведи полный правовой мониторинг уровня L3: общественная проблема...
```

OpenAI указывает, что `SKILL.md` должен содержать метаданные `name` и `description`; Codex первоначально загружает именно имя, описание и путь, а не полный текст. Неявный выбор скилла также опирается на `description`.

Следовательно, эти восемь скиллов нельзя считать надежно обнаруживаемыми или загружаемыми Codex до исправления.

### Исправление

Все двенадцать frontmatter лучше привести к одному формату:

```yaml
---
name: legal-monitoring
description: >-
  Проведи полный правовой мониторинг регулирования: установи общественную
  проблему, цель нормы, механизм реализации, практику, данные, outcome,
  impact, причинность и первопричины. Используй только для комплексной оценки
  регулирования; не запускай для простой правовой справки.
---
```

Folded scalar `>-` устраняет риск синтаксических ошибок и позволяет поддерживать читаемую длину строки.

## P0-2. Валидатор не выполняет настоящий YAML-разбор frontmatter

`validate_workspace.py` содержит собственную функцию `_read_frontmatter`. Она:

1. ищет разделители `---`;
2. берет каждую строку;
3. делит ее по первому двоеточию;
4. удаляет кавычки.

Она **не передает frontmatter в `yaml.safe_load`**. Поэтому строка, невалидная для YAML, успешно принимается внутренним валидатором. Далее тот же самописный результат используется при проверке скиллов.

Именно поэтому одновременно наблюдаются:

- `Status Success` у CI;
- успешная публикация release workflow;
- восемь файлов с явной ошибкой YAML в опубликованном релизе.

Это **дефект quality gate**, а не косметическая проблема.

### Требуемое исправление

Frontmatter должен разбираться реальным YAML-парсером:

```python
raw_frontmatter = "\n".join(lines[1:end])
data = yaml.safe_load(raw_frontmatter)

if not isinstance(data, dict):
    raise ValueError("Frontmatter must be a YAML mapping.")

if set(data) != {"name", "description"}:
    raise ValueError("Expected exactly: name, description.")

if not all(isinstance(data[key], str) for key in ("name", "description")):
    raise ValueError("name and description must be strings.")
```

В тесты необходимо добавить минимум два regression case:

```text
unquoted colon in description → FAIL
folded YAML scalar → PASS
```

После этого release workflow должен использовать тот же валидатор перед сборкой ZIP и перед созданием GitHub Release.

## P0-3. Первичная установка не формирует Git-корень

Runtime README предлагает:

```text
распаковать ZIP
→ установить зависимости
→ validate_workspace
→ check_repo_safety
→ new_matter
→ start_session --create-branch
```

Команды `git init`, первоначального commit и проверки Git-конфигурации в последовательности нет.

При этом `start_session.py` явно выполняет:

```python
git rev-parse --show-toplevel
```

и завершает работу с ошибкой:

```text
ERROR: run from a Git-initialized workspace root.
```

если рабочая папка еще не является Git-репозиторием. Скрипт также требует чистое рабочее дерево для `--create-branch`.

То есть официальный happy path из README для обычного пользователя не является завершенным.

### Исправление

Нужен единый bootstrap:

```bash
python3 scripts/init_workspace.py
```

Он должен:

1. проверить версию Python и наличие Git;
2. проверить структуру и safety profile;
3. выполнить `git init`, если `.git` отсутствует;
4. установить имя основной ветки;
5. предложить режим памяти;
6. создать первоначальный commit только из distributable/managed файлов;
7. не создавать remote автоматически;
8. показать инструкции по trust, hooks и skills;
9. только после этого разрешить `new_matter` и `start_session`.

## P0-4. Не включена процедура доверия к проекту и проверки hooks

Runtime поставляется с `.codex/config.toml`, hooks и command rules. Но проектный `.codex/`-слой полностью игнорируется до тех пор, пока проект не признан доверенным. Project-local hooks дополнительно требуют проверки и доверия к точному хэшу hook definition; для этого предназначена команда `/hooks`.

Следовательно, пользователь может распаковать архив, запустить Codex и полагать, что:

- safety rules активны;
- SessionStart загружает память;
- Stop hook проверяет handoff;
- command rules работают;

хотя в недоверенном проекте эти механизмы могут быть неактивны.

### В README должен появиться обязательный этап

```text
1. Запустить Codex из корня workspace.
2. Проверить источники загруженных инструкций.
3. Признать проект доверенным только после просмотра `.codex/`.
4. Выполнить `/hooks`, проверить и доверить ожидаемые hooks.
5. Выполнить `/skills` и убедиться, что отображаются все 12 скиллов.
6. Проверить доступность ожидаемых специализированных ролей.
7. Только после этого создавать первое реальное дело.
```

# 3. Существенные, но не блокирующие замечания

## P1-1. Противоречие в модели операционной памяти

`memory/README.md` называет `memory/` небольшим **отслеживаемым Git-слоем** и требует локальный `memory:` или `decision:` commit для существенных изменений.

Но runtime `.gitignore` исключает:

```text
memory/CURRENT.md
memory/DECISIONS.md
memory/OPEN_QUESTIONS.md
memory/index.yaml
memory/sessions/*
matters/*
artifacts/*
outputs/*
```

Runtime `AGENTS.md` частично объясняет это: matters и memory исключены до явного решения о хранении. Это безопасная консервативная политика, но пользовательский контракт остается неясным.

### Рекомендация

Bootstrap должен предложить один из трех режимов:

| Режим | Поведение |
|---|---|
| `untracked` | Все дела и изменяемая память локальны и не входят в Git |
| `local-git` | Память и безопасные метаданные отслеживаются только в локальном Git, remote отсутствует |
| `private-approved` | Разрешенные организацией материалы могут попадать в одобренный закрытый remote |

До выбора режима нельзя одновременно обещать пользователю Git-память и игнорировать всю изменяемую память.

# 5. Обязательный план исправления

## Этап 1. Исправить release blockers

1. Привести YAML-frontmatter всех двенадцати скиллов к `>-`.
2. Заменить самописный parser на `yaml.safe_load`.
3. Добавить regression tests для невалидного frontmatter.
4. Добавить `init_workspace.py`.
5. Добавить Git initialization и initial commit в happy path.
6. Добавить project trust, `/hooks` и `/skills` в onboarding.
7. Провести smoke test на конкретной версии Codex.

# Формальное заключение

> Репозиторий `minius_codex_lab` правильно реализует основную архитектурную идею: публичный upstream отделен от пользовательского рабочего пространства; maintainer- и runtime-инструкции разделены; юридические процедуры оформлены как скиллы; специализации оформлены как роли; предусмотрены Git-память, доказательная дисциплина, безопасность и проверяемый юридический документ.
>
> Вместе с тем опубликованный релиз `v1.0.0-beta.1` содержит критические технические дефекты: восемь из двенадцати `SKILL.md` имеют невалидный YAML-frontmatter, действующий валидатор не обнаруживает этот класс ошибок, первичная установка не создает обязательный Git-корень, а фактическая работа Codex skills, hooks, rules и roles не была smoke-tested.
>
> **Начинать массовое внешнее использование `v1.0.0-beta.1` не рекомендуется.** Репозиторий можно использовать как исходную базу разработки и для ограниченного технического пилота исключительно на синтетических данных. После устранения P0-дефектов, публикации `v1.0.0-beta.2` и успешного end-to-end теста проект можно допустить к статусу публичной beta для внешних пользователей.
